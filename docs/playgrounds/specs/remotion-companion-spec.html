Companion feature · v1 pre-implementation · branch feat/remotion-companion

Remotion renders.
The slicer ships.

Make seo-video-slicer the output companion for Remotion and an invokable plugin for Claude Code / Codex: a headless slice CLI, a loop output mode (a new seo-video-slicer.loop.v1 template), slice_video / slice_frames MCP tools, and a Remotion --sequence recipe — all over the primitives that already exist, never mutating the frozen package contract.

Intent · renderer → optimizer → repo

Remotion is the renderer. We are the web-delivery optimizer.

A raw Remotion MP4 is heavy and poor for a hero or scroll moment — bad LCP, autoplay limits, no scroll-scrubbing, no reduced-motion story. The slicer already turns a clip into a Core-Web-Vitals-friendly WebP package with a working, self-contained player. This feature makes that pipeline headless, agent-invokable, and adds a second output mode.

01 / companion

Headless by design

One command (or one MCP call) turns a video or a Remotion --sequence dir into a gated package. Non-interactive. The CLI exits non-zero on a gate failure (CI-ready); the MCP call still succeeds and returns verify.pass=false so the agent keeps the gate detail.

02 / plugin

An MCP tool for agents

slice_video / slice_frames over stdio for Claude Code and Codex. The agent says “render, then slice into a loop” — and it just works.

03 / loop

A second output mode

An auto-advancing rAF player + an animated loop.webp export, as a NEW parallel contract. The frozen scroll package is untouched.

The loop.v1 contract · Phase 0 · define this FIRST

A new parallel contract. The frozen v1 is never touched.

A loop package cannot be a package.v1 manifest: v1 freezes the schema, player.interaction, and the data-template-id to scroll values, and the gate enforces the scroll template-id in G4/G5. So loop gets its own schema + template id, documented in a new sibling CONTRACT-loop.md that leaves the frozen CONTRACT.md byte-identical.

scroll — seo-video-slicer.package.v1 (FROZEN)

  • schema: seo-video-slicer.package.v1
  • player.interaction: scroll
  • data-template-id: seo-video-slicer.scroll.v1
  • player: scroll-progress → frame index
  • gates: G1–G7
  • artifact: frames + index.html

loop — seo-video-slicer.loop-package.v1 (NEW)

  • schema: seo-video-slicer.loop-package.v1
  • player.interaction: loop
  • data-template-id: seo-video-slicer.loop.v1
  • player: time-based rAF → frame index (G4, loop assertions)
  • gates: G1–G7 (adapted) + G8, G9
  • artifact: frames + index.html + loop.webp

The loop package — folder + manifest

<slug>-loop/ ├── frames/ │ ├── frame_000.webp # zero-padded, contiguous (shared with scroll) │ └── … frame_NNN.webp ├── index.html # auto-advancing rAF player · data-template-id=loop.v1 ├── loop.webp # NEW · animated WebP · Pillow save_all · loop=0 ├── manifest.json # loop-package.v1 + loop{} block └── README.md # two-tier embed: <img> loop.webp | canvas index.html manifest.loop = { fps: 12, duration_s: 2.5, webp: "loop.webp", webp_sha256: "<hash of loop.webp>", // gated by G9 loop_count: 0 // infinite } // the loop block is OUTSIDE the fingerprint

Fingerprint — reused verbatim

The frozen recipe fingerprint(frameBasenames, gsapUrlOrEmpty, templateId) is reused character-for-character. For loop: same frames, gsap="" (pure rAF, no GSAP), templateId="seo-video-slicer.loop.v1" → a distinct valid sha256 for free, recipe byte-identical between builder and verifier. fingerprint() stays exactly two copies; no third copy, no new recipe.

The gap the fingerprint can't see — and how G8/G9 close it

fps is a build-time parameter that bakes loop.webp's bytes. Two loops that differ only in fps share the same 3-input fingerprint — so a re-encoded or swapped loop.webp would be invisible to G5. We do not fold fps into the frozen recipe; we add two new gates. Drag the slider:

fps = 12 · frame duration 83 ms
fingerprint(frames, "", loop.v1) 3-input recipe — unchanged by fps SAME · G5 passes
loop.webp content sha256 — tracks fps
manifest.loop.webp_sha256 — (recorded at build: fps 12) recorded once
G8 · Σ ANMF dur == frames × perFrameMs 2490 ms == 2490 ms PASS
G9 · sha(loop.webp) == manifest match PASS

Illustrative sha (a deterministic stand-in, not a real WebP encode). The point is structural: the 3-input fingerprint is fps-blind, so loop integrity needs dedicated gates. G9 re-hashes the baked loop.webp (catches re-encoded bytes); G8 sums the ANMF frame-durations and binds the total to frames.count × perFrameMs (catches a manifest-fps-only edit with the bytes untouched). Together they make fps ↔ bytes ↔ manifest mutually consistent — the frozen recipe is never touched. Sum-based (not per-frame) because libwebp coalesces byte-identical consecutive frames into fewer ANMF with summed durations; perFrameMs = floor(1000/fps + 0.5) is pinned identically in the Python encoder and the Node verifier (banker's-rounding parity bug at fps 16).

Loop gates — G8/G9 new, G4/G7 adapted

G8loop.webp structure + fps binding (NEW). Real animated WebP — RIFF/WEBP, VP8X with the ANIM flag, an ANIM chunk, and 1 ≤ ANMF count ≤ frames.count (libwebp coalesces identical consecutive frames). Cadence bound by the SUM: Σ(ANMF 24-bit durations) == frames.count × perFrameMs, perFrameMs = floor(1000/fps + 0.5) pinned identically in Python & Node. node:fs, zero-dep.
G9loop.webp integrity (NEW). sha256(loop.webp) == manifest.loop.webp_sha256 (node:crypto). A re-encode changes the bytes → FAIL.
G4loop technique check (adapted, presence gate). cover-fit, DPR, onerror preload, reduced-motion block, data-template-id=loop.v1, and the positive time markers requestAnimationFrame + performance.now(). The time-driven loop is additionally proven in-browser in Phase 0 (regex can't prove a formula).
G7weight (adapted, re-measured from disk). Hard-fail stays frames.count > 200 (and < 1). loop.webp counts toward total_bytes; oversized loop.webp is a soft warning, never a fail.

Two-tier delivery (state it, or it reads as redundant weight)

  • <img src="loop.webp" loading="lazy"> — simplest possible embed.
  • Ignores prefers-reduced-motion (it's an <img>).

index.html (canvas + frames) — the instrument

  • DPR-crisp, controllable, honors reduced-motion (static hero frame).

Acceptance — the 5-star bar

A1Contract safety. Scroll output byte-identical — a golden-package byte-diff (every output file, excl. per-build id/created_at; fingerprint pinned within it) fails on any scroll drift; CONTRACT.md byte-unchanged. (The .mjs files legitimately gain a loop branch.)
A2Loop gates. Build loop → verify passes; corrupt frame / sha / ANMF count / ANMF duration → matching gate fails; editing manifest.loop.fps alone fails G8.
A3Loop player. Loops offline; zero network; reduced-motion shows a static frame.
A4CLI. video + frames, scroll + loop produce passing packages; non-zero exit on gate fail; --json shape.
A5MCP + Remotion. Tools return passing packages over stdio; examples/remotion/ renders & slices via make on a clean clone.
A6CI + docs. make test + CI green; README/recipe at the existing polish bar; no stale context.

Non-goals

×Mutating seo-video-slicer.package.v1 or its scroll code path.
×An in-app animation generator, local LLM, or chat (inherited non-goals).
×A loop video editor / GIF studio. Loop = one player + one loop.webp, gated.
×Vendoring ffmpeg/node. They stay system prerequisites; mcp is an optional extra.
×Folding fps/loop.webp into the frozen fingerprint recipe (G8/G9 instead).
×A preview UI in the CLI/MCP path. They are plumbing.
